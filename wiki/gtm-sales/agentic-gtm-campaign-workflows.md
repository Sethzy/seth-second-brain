## Overview

This bundle adds a practical GTM pattern: agents are moving from list building into full campaign assembly. The repeated ingredients are richer company filters, agent-accessible enrichment APIs, generated personalized landing pages, outbound copy, sequencing, and suppression/dedupe logic.

The most concrete examples are Prospeo's expanded filters and MCP access, Clay research feeding Flint AI personalized landing pages, and a list of Claude Code-ready GTM tools that either feed agents with data or let agents build workflows.

Sean Wilson's six-campaign post gives a useful operating checklist: competitor displacement, case-study lookalikes, web visitors, signal/trigger plays, evergreen social/champion/job-change plays, and macro ICP volume. Ayman Al-Abdullah's Architect Mode post adds the post-call learning loop: transcribe calls, score them, identify phrases correlated with closed deals, update scripts, and surface coaching needs in near real time.

The Acme bundle adds a daily operating cadence: every key account should have multiple sales angles, a first-draft deck/proposal, and one thoughtful researched outreach per day. Shann Holmberg's proposal-agent post suggests compressing proposal turnaround by running pre-call diligence plus meeting transcript context into a live proposal draft. Actively's homepage is the market analogue: per-account agents that work accounts continuously and guide sellers on next actions.

Roman's Claude MCP + LinkedIn article is another version of the same loop, with Gojiberry as the execution layer: describe the ICP, find high-intent leads, enrich with job/hiring/engagement signals, generate personalized LinkedIn messages, launch campaigns, and review weekly campaign performance from the agent.

The Vercel templates add implementation skeletons for two GTM-adjacent agents. The Call Summary Agent turns Gong-style call transcripts into structured summaries, objections, tasks, and insights using Workflow DevKit plus Vercel Sandbox for file exploration. The Lead Agent starts from a contact-sales form, runs lead research, qualifies the lead with structured output, drafts an email, and asks for Slack approval before sending.

Prathit Joshi's GTM agent post reinforces the operator pain point: much GTM work is opening tabs and copying data across systems. The claimed workflow is parallel independent task threads that scan Reddit trends, find leads, inspect PostHog analytics, and connect to a large tool surface. Treat the repo/link behind the short URL as a follow-up capture target before relying on implementation details.

Eric Siu's AI Marketing Skills repo is the skill-pack version of these GTM loops. Instead of a single prompt, it packages categories like Growth Engine, Sales Pipeline, Content Ops, Outbound Engine, SEO Ops, Finance Ops, Revenue Intelligence, Conversion Ops, Podcast Ops, Team Ops, Sales Playbook, Autoresearch, Deck Generator, YouTube Competitive Analysis, and X Long-Form plus Humanizer. The useful framing for Seth is that each skill should be treated as an operational workflow with data inputs, scoring, tool calls, and review gates, not as generic copywriting help.

Eric's launch post sharpens the revenue standard for skill packs: the workflows should resurrect deals, turn visitors into pipeline, run marketing experiments, create expert-panel content, and even recover money through finance ops. In other words, a GTM skill should produce an auditable revenue action or decision, not just a prettier artifact.

The Desktop Archive GTM notes add a broad saved-links backlog: AI coworkers for Series A/B GTM teams, AI CRM examples, Salesforce/AI sales startup links, LangChain's GTM agent, directory SEO/backlinks, LinkedIn outbound, Google ads, UGC playbooks, web automation stacks, NotebookLM/content ideas, Pencil/Fyxer-style productivity references, and GEO SEO skill leads. These are mostly Seth-intent leads rather than complete evidence, so promote them only after capturing the underlying source.

Claude SEO makes the SEO Ops lane concrete: a specialist skill can run technical, content, schema, local, e-commerce, international, programmatic, backlink, Search Console/PageSpeed/CrUX, and GEO audits from a command surface, then produce prioritized, falsifiable recommendations. For Acme/eGiro, the immediate use is not generic SEO advice; it is repeatable site audits, comparison-page planning, programmatic SEO checks, and AI-search citability checks before publishing.

The Sunder sync is now the broader local evidence lane for GTM material: LangChain-style GTM agents, SalesCraft/outreach references, local TAM scrape outputs, LinkedIn/X outreach notes, website/SEO experiments, and old campaign/data exports. It should be used as a retrieval index for campaign ideas and source files, with raw captures checked before an item becomes a playbook.

The GTM Context OS repo is the clearest template for turning these campaign ideas into a repo-native operating system. Its useful move is to bind context and operations to the same git workspace: `context.md` as durable company strategy, `demand/` as evidence and PULL analysis, `status.md` as the operational log, optional modules for segments/messaging/campaigns/engine/content/scripts/workflows, and agent rules that require campaigns and messaging to link back to demand evidence. It also gives Seth a directly reusable demand-scoring rubric: Project, Unavoidable, Looking, Lacking, with transcripts or detailed buyer language required before a lead becomes "demand" rather than generic benefit.

## Enterprise Sales Workflow Addendum

The 2026-06-16 Salescraft, X profile, and cross-repo captures sharpen the difference between campaign automation and enterprise selling. The workflow should not start with "send more." It should start with a finite account universe, current trigger evidence, relationship-path search, and a reason the timing is unusually good.

The Salescraft/Patrick/Eric corpus belongs in the campaign layer as reusable playbook material: lists of hooks, objection frames, follow-up patterns, founder-led outreach lessons, and examples of social proof. It is useful for draft generation, but each outbound claim still needs account-specific evidence from the target account, buyer language, or a nearby customer/proof point.

The four X profile sweeps add operating patterns rather than canonical facts:

- Alfie Carter: package sales workflows as reusable skills/MCPs, especially LinkedIn/navigation and account-research skills that can be shared between agents.
- Fivos Aresti: use ABM signal stacks that combine first-party, second-party, and third-party intent, then route different signals into different sales motions.
- Chris Pisarski: build campaign workflows around practical speed and buyer context: Slack alerts, phone-number capture on demo forms, warm intro pathfinding, pre-call briefs, budget discovery, and fast follow-up.
- Brian LaManna: separate whale hunting from generic outbound. Tier 1 accounts need onsite/events/references/value-engineering motions, customer-story mining, and closed-lost prospecting.

For Acme/eGiro, the compiled workflow is: pick a named account, verify it is inside the regulated TAM, check fresh triggers, map stakeholders and relationship paths, produce five angle drafts, select the strongest proof-proximity angle, create a deck or one-pager, and only then draft a single outbound message for review.

## Backdated X Sweep Addendum

The second profile page adds three workflow-level ideas:

- GTM skills need shared context. Alfie's older posts are noisy, but the reusable point is that a skill library should bootstrap brand, voice, ICP, tools, and references once, then let task-specific skills cross-reference that shared context. Otherwise "skills" collapse back into isolated prompts.
- Content should produce account signals. Fivos repeatedly frames LinkedIn as a system for audience warming, engagement capture, contact capture, CRM updates, and outbound follow-up. The durable workflow is content -> engagement signal -> enrichment -> CRM/account update -> human-reviewed outreach or nurture.
- Personalized generated assets can be the outbound asset. Fivos's microsite/deck pattern is a more concrete version of account-specific landing pages: use account research, company colors, ICP findings, and a custom proposal to create a reviewable asset before or after a warm interaction.

The role shape is the "GTM engineer": TAM mapping, signal tracking, enrichment, scoring, routing, automated outbound, inbound orchestration, CRM/reporting, and campaign iteration. For Acme/eGiro this should be a function of the operating system, not a separate toy automation lane.

## Key Ideas

- Prospeo-style filters make ICP definition more semantic: AI lookalikes, business model, pricing model, website keywords, AI attributes, compliance/certs, traffic, executive events, news, integrations, customers, Google discovery, and target-customer filters.
- Personalized outbound can include a generated asset, not just a personalized sentence. The Flint/Clay pattern researches an account, generates a prospect-specific landing page, and inserts the page URL into outbound.
- Buying-window triggers matter more than static personas: new executives, competitor mentions, hiring signals, funding, tech installs, and fresh website/page changes.
- The GTM stack is splitting into "feed" tools and "build" tools. Feed tools expose data APIs to agents; build tools orchestrate campaigns, sequencing, governance, and reply handling.
- Campaign automation should preserve dedupe, suppression lists, geo/segment splits, cost tracking, source provenance, and human approval before sending.
- Always-on outbound should be portfolio-based, not one campaign: competitor, case-study, web-visitor, trigger/signal, evergreen, and macro ICP motions each need different data feeds and copy logic.
- Sales coaching can become a data loop when call transcripts, objections, winning phrases, rep behavior, and CRM outcomes are joined rather than reviewed manually after the fact.
- AI-generated flyers, landing pages, slides, and website revamps can become campaign assets, but they need source provenance, brand checks, and approval before use.
- Account agents should produce concrete artifacts: next action, evidence, angle, draft email/DM, proposal/deck, and CRM update.
- Proposal and deck agents need meeting transcript, account research, pain-point cheat sheet, pricing/onboarding constraints, and brand system context.
- LinkedIn outreach agents need an execution layer and confirmation gate; the model should not directly send messages without preview, approval, and campaign/audit tracking.
- Lead-processing and call-summary agents should share the same governance pattern: durable workflow, structured outputs, source files in a sandbox, Slack review, and explicit approval before CRM/email side effects.
- GTM agents should be designed around parallel research threads, because scanning communities, analytics, lead lists, and CRM context are naturally independent subtasks that can merge into one reviewed recommendation.
- Marketing/sales skill packs are most valuable when they encode repeatable operating loops: deal resurrection, ICP learning, RB2B routing, expert-panel scoring, content pipelines, call intelligence, and deck generation.
- Revenue skill packs should be judged by concrete actions and recovered value: pipeline created, deals revived, experiments shipped, costs recovered, and follow-up decisions logged.
- SEO workflows should be treated like recurring GTM operations: crawl, measure, compare, generate a prioritized action plan, ship fixes, and rerun drift monitoring.
- The Archive batch suggests a separate "GTM watchlist" lane may be useful for weak leads that are not ready to become system design.
- A GTM context repo should be both evidence base and execution surface: durable strategy in `context.md`, demand evidence in `demand/`, campaign operations in modules, scripts/workflows for APIs, and `status.md` for session history.
- PULL scoring is a useful gate before creating segments, messaging, or outbound campaigns: if there is no active project, unavoidable timing, looking behavior, and lacking alternative, the workflow should treat the account as benefit/lead-nurture rather than demand.
- Agent-authored GTM systems need provenance tags and freshness dates, because CRM fields, pricing, headcount, and "currently" claims can remain verified but become stale.
- Enterprise campaign agents should track the distinction between generic benefit, active demand, relationship path, and proof proximity before recommending a channel or cadence.
- Content-led campaigns should be wired to CRM/account intelligence: the system should know which posts generated the engagement, which people engaged, whether they match ICP, and which route each signal deserves.
- GTM skill libraries should start from a shared company/brand/ICP context layer so outreach, research, deck, and landing-page skills do not reinvent positioning on every run.

## Campaign Motions To Test

- Competitor displacement: detect competitor usage, then generate comparison-specific copy.
- Case-study lookalikes: map closed-won stories to similar companies and route the matched proof point into outreach.
- Web visitors: deanonymize traffic, enrich, qualify, personalize to visited pages, and follow up quickly.
- Signals and triggers: hiring, funding, job changes, tech changes, news, executive events, and competitor mentions.
- Evergreen: social engagement, champion follows, job alerts, and community signals.
- Macro ICP: lower-signal volume with strong copy and useful lead magnets.
