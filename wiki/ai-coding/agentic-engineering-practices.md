## Overview

Agentic engineering is less about installing the newest harness and more about controlling context, defining task contracts, and tightening verification loops. The sysls article argues for a barebones setup, precise context routing, neutral prompts, explicit stop conditions, and ongoing cleanup of rules and skills. Alok Bishoyi's autoresearch article shows the next step: using agents to optimize the harness itself by experimenting on skills, scoring against a benchmark, and keeping an audit trail.

The shared thesis is that the wrapper around the model is now a first-class engineering surface. Skills, rules, task contracts, benchmark gates, and context boundaries can improve faster than model weights because they are editable text and local workflow design.

Z.ai's GLM-5.2 launch note is a fresh model-layer watch item for this page: it positions an MIT-licensed open-weight model around coding, agentic tasks, long-horizon work, a 1M context window, and selectable reasoning effort. Treat this as a lead for future model evaluation rather than a settled benchmark claim; the durable point is that open-weight models are now explicitly competing on the same harness-relevant dimensions as closed coding-agent models: long context, reasoning budget, cost, and agentic-coding behavior.

Vox's prompt-shaping note and screenshot templates make the operational point sharper: Claude Code/Codex should not be treated like a junior task runner. One-line imperatives such as "do this," "write code," or "fix this bug" waste the model's planning and reasoning capacity. The eight captured templates repeatedly specify role, goal, deliverables, architecture/edge cases, verification, and completion standard: a general end-to-end goal wrapper, a parallel big-task goal, production-grade build prompt, unfamiliar-repo refactor prompt, senior debugging prompt, performance-optimization prompt, production UI-component prompt, and publishable API prompt. The pattern is less about magic wording than about making the acceptance contract explicit enough for the agent to plan, verify, review, and know when to stop.

The AI Engineer talks batch turns this into a broader state-of-the-art view. Matt Pocock supplies the anti-vibes foundation: software fundamentals, TDD, shared language, deep modules, and interface-first design matter more because agents amplify both good and bad codebase structure. OpenAI's harness-engineering talk pushes the other side: when implementation capacity is abundant, the scarce resources are human attention, model attention, and context, so teams should invest in agent-readable requirements, QA plans, reviewer agents, lint/test prompts, and recurring-failure guardrails. Kiro's spec-driven development talk reconciles these positions by treating specs as requirements/design/execution artifacts plus verification tooling, not as a substitute for understanding code.

Denys Linkov's AI-team-structure talk adds the org-design version of the same argument. The "AI team" should be designed around the company's bottleneck and business type, not around trendy titles. Many teams need generalists who combine model training literacy, serving/platform judgment, domain expertise, product requirements, business-case thinking, evaluation writing, and customer-facing feedback loops before they need top-lab researchers.

The YC Root Access context-engineering talks add a useful retrieval/compaction discipline. Chroma's frame is that context engineering is simply deciding what goes into the context window: gather relevant candidates broadly, glean away noise, and optimize what remains. Dex Horthy's advanced coding-agent version turns that into a workflow: keep context under control, use subagents for search, write research files with filenames/line numbers, write explicit plans with verification steps, and review those artifacts before implementation. The expensive mistakes happen upstream: bad research can create thousands of bad lines of code.

Peter Wang's vertical-agent article sharpens the context-engineering side: a good domain agent is a faithful compression of its task distribution. Treat system prompts, tools, skills, curated docs, and raw references as one context hierarchy. L1 is the always-resident hot path for bread-and-butter tasks; L2 is curated, on-demand English specs and deferred tools; L3 is the complete raw substrate plus a small skill that teaches the agent how to mine it. The target is not "put everything in context," but minimize context spent per task while preserving accuracy.

The Codex-maxxing and implementation-notes sources add a practical rule: make the loop visible. The agent should not only change files; it should leave behind a reviewable trace of design decisions, deviations from spec, tradeoffs, and unresolved questions.

Zach Lloyd's spec-driven development loop turns task contracts into reviewable repo artifacts: write a product spec for user behavior and invariants, write a tech spec for implementation strategy and code locations, check both into the implementation PR, implement against them, then run a validation skill that compares the PR against the specs and surfaces mismatches. For UX changes, a computer-use validation pass in a sandbox becomes the final acceptance gate.

Karpathy and David Breunig push this idea to its edge. Karpathy describes spec-driven development as the limit of the imperative-to-declarative transition: the human states the desired behavior rather than hand-writing every operational step. Breunig's `whenwords` experiment makes that concrete by shipping a "library" as `SPEC.md`, `tests.yaml`, and installation instructions for an agent to implement in the user's language and repo. The important caveat is that spec-only software is plausible for tight utilities with simple standards and robust tests, but weak for foundations that need performance, complicated test matrices, support, security updates, and community-maintained reference implementations.

EveryInc Compound Engineering and obra Superpowers make this loop more operational. Compound Engineering frames agentic coding as a compounding cycle: strategy, ideation, brainstorm, plan, work, debug, review, product pulse, and compound learning. Its useful claim is that planning and review are the leverage surface, not bureaucracy: a better brainstorm shrinks implementation, a better review captures reusable patterns, and a compound note prevents future agents from relearning the same local lesson. Superpowers is a stricter software-development methodology for agents: before code, brainstorm and get design signoff; before work, create a worktree and a bite-sized plan; during implementation, enforce red/green TDD, subagent-driven task execution, spec compliance review, code-quality review, and verification before completion.

The newest loop-engineering posts converge on a sharper operating model. Addy Osmani defines loop engineering as designing the system that prompts, checks, remembers, and schedules agents instead of manually prompting each turn. His loop stack is automation, worktrees, skills, plugins/connectors, subagents, and external memory. Lance Martin adds that strong models should often be run inside self-correction loops with independent grader/verifier contexts and cross-session memory, not simply steered turn by turn. Sairahul's loops post says the same thing more economically: a loop has discovery, planning, execution, verification, and iteration, but the practical version needs token budgets, closed scopes, worktree isolation, skills, connectors, subagents, and a state file.

The harness-engineering captures make the layer below the loop concrete. Sairahul's harness piece defines the harness as everything around the model: constraints, feedback loops, docs, tool permissions, context, progress files, planner/evaluator splits, and codebase legibility. Sydney Runkle's custom-harness article maps the same idea to middleware: deterministic business logic, tool lifecycle management, custom state, stream handling, memory, context engineering, sandboxing, guardrails, human-in-the-loop checkpoints, and observability. The practical rule is: agent = model + harness, and most reliability gains come from improving the harness.

The Tech with Mak production-architecture screenshot is a compact repo-shape version of the same thesis for RAG/AI apps: a demo can be one file, but a production system separates entry/config/models, retrieval components, service-level RAG/cache/memory/rewrite/routing logic, versioned prompt registries, agent/source-selection logic, tool definitions, security guards, evals, observability, data/index config, scripts, UI, tests, docs, and agent-readable project context. Treat it as a checklist pattern, not a framework mandate: the point is explicit ownership boundaries for reliability surfaces that are often hidden inside a notebook or single route handler.

LangChain's Deep Agents overview is a useful official reference architecture for that harness layer. It keeps the same core tool-calling loop as ordinary agents, but packages the recurring reliability components: task planning through `write_todos`, context compression via a virtual filesystem, custom functions/LangChain tools/MCP tools, pluggable filesystem backends with permission rules, shell execution through local or sandbox backends, JavaScript interpreters for data transformation, a `task` tool for isolated subagents, event streaming including subagent streams, long-term memory through LangGraph stores, human approval interrupts, reusable skills, and opinionated prompts that teach planning, verification, and context management. In other words, it turns many of the captured blog-post patterns into a library surface.

OpenAI's `codex-plugin-cc` README adds a concrete cross-model review pattern: Claude Code can call Codex for normal read-only review, a steerable adversarial review, background rescue tasks, and an optional review gate. The key concept is not just "another reviewer," but model diversity as a review primitive. A same-model reviewer can still share the maker's priors, training distribution, and blind spots; a Codex review from inside a Claude workflow creates a stronger adversarial pair because the critic is a different model family and runtime. The plugin's review gate makes this loop explicit by using a Claude `Stop` hook to run targeted Codex review and block completion when Codex finds issues, with the important warning that this can become a long-running Claude/Codex loop and should be monitored.

The walkinglabs Awesome Harness Engineering set turns the scattered reading list into a more complete harness-engineering map. The curated subset Seth asked to capture reinforces that "harness" is not one product category. It is the control plane across context, memory, task state, tool design, constraints, specs, evaluation, observability, and runtime safety. The important new synthesis is that reliable agents need three layers at once:

1. **Work definition:** specs, `AGENTS.md`/`CLAUDE.md`, task state, reference applications, and brownfield cleanup make the codebase itself easier for agents to operate inside.
2. **Runtime control:** sandboxing, permission policy, controlled code execution, tool interface design, pause/resume, retries, concurrency, and prompt-injection defenses bound what the agent can do.
3. **Measurement:** trace grading, skill evals, OpenTelemetry conventions, AgentOps/agenttrace-style session review, infrastructure-noise analysis, and Deep Agents evals make behavior inspectable and regressions visible.

The remaining walkinglabs sections add the benchmark and implementation map. The useful move is not to memorize every leaderboard, but to know what each class of benchmark stresses: terminal agents, web agents, OS/computer-use agents, MCP/tool-use agents, multi-turn policy agents, security agents, and long-horizon software-evolution agents. The runtime/reference side shows where harness ideas become inspectable code: Deep Agents, SWE-agent, SWE-ReX, AgentKit, browser harnesses, worktree/memory harnesses, Harbor, Harness Evolver, Ralph loops, skill marketplaces, and universal app/desktop connectors. This gives Seth a vocabulary for comparing agent systems by evaluation surface and runtime primitive rather than by brand.

The strongest practitioner takeaway from the new captures is that harness engineering and software engineering are now overlapping disciplines. Thoughtworks' harness/context/quality/reference-app articles say agents need architectural constraints and entropy control, not just prompts. Anthropic's harness/context/tool/sandboxing/evals articles say long-running agents need initializer agents, task artifacts, self-verification, tool boundaries, sandbox policy, and evals that account for many valid trajectories. OpenHands adds the adversarial/safety side: prompt-injection defenses, skill baselines, deterministic verifiers, and trace review. OpenTelemetry and AgentOps add the observability vocabulary: if an agent acts through tools, files, browsers, APIs, and persistent sessions, the trace must describe those actions, not only the final answer.

The Shopify/Claude Code and convergence posts add product-shape evidence. zodchiii's Shopify capture describes parallel agents, critique loops, an internal LLM proxy, MCP access to real platform docs/schemas, shared `CLAUDE.md`, strategy-first validation, and guardrails that allow read/write/test/lint/commit while blocking irreversible actions. Nicholas Charrière argues that multiple products are converging on the same shape: trusted workflow position, proprietary context, and short observation-to-improvement loops matter more than a single chat surface.

Matt Stockton's post names the next layer as harness engineering: companies can use Codex or Claude Cowork as the main harness, attach internal data sources, and expose repeated work through slash-command skills. For proprietary data, his recipe is FastMCP Python wrappers around API calls, deployed on Modal, with Auth0 for authentication. Ashpreet Bedi's Scout article adds the context architecture: instead of dumping every tool into the main agent, put context providers between the agent and each source so the main agent sees a small `query_<source>` / `update_<source>` surface while a sub-agent owns source quirks.

Kyle Jeong's Browserbase `bb` article makes the production architecture more concrete: separate the agent harness from compute, run each session in an isolated cloud sandbox, pre-warm snapshots with repos/tools, expose a tiny tool surface, route all external access through a credential-brokering proxy, and hard-scope permissions per invocation. Seth's attached `claude --dangerously-skip-permissions` note should be treated as a speed lever only inside this kind of disposable, tightly scoped environment; it is not a default for local machines or broad credentials.

Sebastian Raschka's coding-agent components post is a useful conceptual map for the same architecture: coding agents are not just an LLM call, but a bundle of repo context, tool use, memory, and delegation. Treat it as a follow-up reading target for the underlying building blocks behind Codex/Claude/OpenCode-style systems.

Romain Huet's Codex Security post and the OpenAI setup docs add a security-specific loop for the sandbox discussion. Codex Security runs against GitHub repositories connected through Codex Cloud, requires an environment, scans commits from newest backward across a selected history window, generates a threat model, then lets the team review recommended/all findings and create remediation PRs. The useful pattern for Seth's "G stack" sandbox thinking is explicit context control: repo, branch, environment, history window, threat model, findings, validation output, and PR generation are separate levers.

Ryan Carson's Devin post is the counterweight to custom harness enthusiasm: he describes a Linear-triggered `land` playbook that runs CI and merges green PRs, then argues that many teams should use funded platforms instead of building their own code factory. Ramp Inspect is the opposite but more rigorous case: if a team does build its own background agent, it needs the full stack of sandboxed VMs, repository images and snapshots, fast time-to-first-token, telemetry/feature flag/test access, visual verification, multiple clients, PR attribution, and org-level usage metrics.

The Cognition Devin articles make the managed-platform case much more concrete. ## Key Ideas

- Context is the scarce resource. Keep global instructions small and route agents to only the rules, files, and skills relevant to the task.
- Context engineering is gather and glean: maximize recall across possible sources, then maximize precision by removing distracting tokens before the model reasons.
- Model choice is now part of harness design: open-weight releases such as GLM-5.2 should be evaluated on coding-agent behavior, long-context reliability, reasoning-effort controls, cost, license, and fit with the surrounding tool/runtime layer, not just generic chat quality.
- For large codebase work, research and plan artifacts are review surfaces. Reviewing them is often higher leverage than reading every generated line.
- AI coding should be evaluated as an end-to-end SDLC loop: requirements, design, implementation, tests/evals, review, release, production learning, and skill/doc updates.
- AI team design should be evaluated as an end-to-end capability loop too: define use cases, integrate product, find data, test workflows, build interfaces, measure ROI, sell value, and keep humans accountable.
- Specs are control surfaces for intent and verification, not permission to ignore code quality, architecture, or feedback loops.
- Spec-driven development is the declarative edge of AI coding: behavior, tests, constraints, and install instructions become the reusable artifact, while code may be generated into the local environment.
- Spec-only code works best for small, well-specified utilities. It is risky for foundations that depend on performance, supportability, security updates, interoperability, and community bug-fixing.
- The strongest spec-driven loop turns requirements into acceptance criteria, vertical slices, property/tests/evals, reviewer prompts, and production-learning artifacts.
- Vertical agents should be designed like a memory hierarchy: put the common task distribution in tiny resident L1 context, expose important occasional capabilities through one-step L2 specs/tools, and keep rare long-tail capability reachable through searchable L3 raw references.
- Tool count is itself context cost. A single expressive execution surface can outperform many overlapping narrow tools when the domain API is available through code and well-documented tiers.
- Common operations deserve disproportionate engineering: token-compressed reads, structured diffs, consequence reporting, lint-style feedback, and free adjacent context all increase accuracy on the steep part of the task curve.
- Separate research from implementation when the implementation choice is unknown; let one context decide the approach and a fresh context execute it.
- Avoid outcome-biased prompts when auditing. Ask for findings or trace behavior rather than insisting there must be a bug.
- Define the end of the task explicitly with tests, screenshots, verification steps, or a task contract.
- Prompt senior coding agents with goals, context, constraints, deliverables, verification expectations, and completion standards; one-line imperatives waste the agent's planning ability and hide the acceptance criteria.
- Reusable prompt templates should encode a role, the work surface, edge cases, expected artifacts, review loop, and stop condition. The Vox templates are useful retrieval examples for big tasks, refactors, debugging, performance, UI components, and APIs.
- For larger feature work, prefer checked-in specs over transient prompt context: `PRODUCT.md` for user-facing behavior, `TECH.md` for implementation strategy, and a validation pass that compares the diff against both.
- Treat planning/review/learning as compounding engineering assets. Requirements docs, plans, review findings, product pulse reports, and compound notes should make the next unit of work easier.
- For autonomous implementation, use stricter process gates: approved design, isolated worktree, tiny tasks, red/green TDD, spec-compliance review, code-quality review, and verification before completion.
- Adversarial review is strongest when the checker is not just a fresh context, but a different model/runtime. Different model priors expose different blind spots.
- Cross-model review loops should be read-only by default, steerable around risks, and monitored when wired into stop hooks because they can burn usage in long-running maker/checker cycles.
- Loop engineering moves the human from turn-by-turn prompting to designing the system that discovers work, plans, executes, verifies, remembers, and schedules the next run.
- A practical coding loop needs automation, worktrees, skills, connectors, subagents, and external memory; otherwise it is just a long prompt with higher token burn.
- Harness engineering is the reliability layer around the model: context delivery, middleware, tools, permissions, state, sandboxes, feedback, observability, and human checkpoints.
- Production AI apps should expose reliability boundaries in the repo shape: retrieval, services, prompts, agents/tools, security, evals, observability, data/index config, tests, docs, and coding-agent context should be inspectable as separate surfaces rather than buried inside one demo file.
- Deep Agents is a concrete harness pattern to study: planning tool, filesystem-backed context, MCP/tool connections, shell/sandbox execution, subagent isolation, streams, memory, permissions, human approval, skills, and smart prompts.
- The walkinglabs Awesome Harness Engineering set is now the retrieval router for the harness canon: foundations, context/memory/state, safe autonomy, agent files/spec workflows, and eval/observability resources.
- The full walkinglabs repo is also indexed for benchmark/runtime lookup. Use it to route questions about Terminal-Bench, SWE-bench, WebArena, OSWorld, MCP benchmarks, security benchmarks, Deep Agents, SWE-agent, SWE-ReX, browser harnesses, Harbor, Harness Evolver, Ralph loops, and portable skills.
- Brownfield AI work needs codebase hygiene as harness work: layered repo instructions, ratcheted hooks, baselined lint, feature-folder refactors, and reference applications reduce agent entropy before implementation starts.
- Runtime control is a product surface: sandbox policy, controlled code execution, tool design, pause/resume state, retries, concurrency, credential boundaries, and prompt-injection analyzers are harness primitives.
- Agent observability should be portable and trace-level: skill evals, trace grading, OpenTelemetry semantic conventions, cost/session replay, health gates, and infrastructure-noise tracking are part of the engineering loop.
- Evaluate skills and harness changes with no-skill baselines, deterministic verifiers, trace review, realistic task distributions, and awareness that runtime configuration can move benchmark scores.
- Prefer a new session per well-specified contract over one long-running session that accumulates unrelated context.
- Rules encode preferences and constraints; skills encode recurring recipes.
- Self-improvement loops become tractable because the parameter space is markdown: skills, prompts, configs, and harness files can be edited, benchmarked, and audited.
- Optimization loops need gates. Without regression tests, held-out slices, or no-cheating audits, they can game the metric.
- Long-running or ambiguous implementations should maintain a scratch-log or `implementation-notes` artifact while work is happening, not only a final summary.
- Review surfaces matter. If the artifact can be inspected visually or structurally in the same app, the human can steer the agent without resetting context.
- Harness engineering has two separable jobs: choose the agent surface, then design the data/tool/context layer that makes the surface useful.
- FastMCP-style wrappers are useful when a proprietary API has no existing MCP server, but tool definitions must carry the right context and auth boundaries.
- Context providers reduce tool pollution by hiding source-specific quirks behind a small query/update interface and letting sub-agents handle pagination, identity lookup, thread traversal, and other source mechanics.
- Navigation over search is a useful design principle for company context: prefer live source traversal with real citations when stale vector chunks would mislead the agent.
- Production agents need structural safety, not vibes: scoped tokens, proxy-enforced service/method allowlists, read-only roles, domain allowlists, and disposable sandboxes.
- Permission auto-mode or `--dangerously-skip-permissions` belongs only behind compensating controls: ephemeral sandbox, no long-lived secrets, narrow filesystem, scoped network, and audit logs.
- Coding-agent evaluation should separate model capability from harness components: repo context, tool surface, memory, delegation/subagents, sandbox, permissions, and verification.
- Security agents need a living threat model, not just a scanner. Repo scans become more useful when architecture, trust boundaries, business context, and current priorities are editable inputs.
- Buy-versus-build is a first-order harness decision. Devin-style managed platforms are attractive when orchestration itself is not the product; Ramp Inspect-style internal agents make sense when proprietary context, verification, and workflows are the advantage.
- Managed-agent platforms compound when review comments, CI/lint failures, security findings, and dependency bot output can trigger automatic fix sessions before human review.
- Prompting autonomous coding agents is still engineering work: describe the approach, starting context, expected checks, likely failure modes, and verification mechanisms.
- Hosted coding agents should optimize the whole loop, not just code generation: warm sandbox, repo sync, read/edit gates, tool context, visual verification, PR creation, review, merge, and usage analytics.
- Local multi-agent orchestration can start with terminal multiplexing. A `tmux-bridge`-style CLI makes panes into addressable work surfaces, but it needs explicit labels, narrow handoffs, and caution around privileged shells.
- Use-case galleries are reusable task-contract libraries. Each starter prompt can be mined for objective shape, context needed, verification expectation, and handoff pattern.
- Link-only engineering notes should become a follow-up queue: capture the underlying article/repo/post first, then promote durable claims into this page.

## My Take

For Seth Second Brain, this points toward two maintenance loops: a small rule/router layer that tells Codex where to find context, and periodic skill/wiki cleanup that removes contradictions before the corpus becomes prompt bloat. For coding work, Seth's default stack should combine Matt Pocock for compact engineering fundamentals, Compound Engineering for compounding plan/review/learning loops, and Superpowers for strict autonomous implementation gates. The most useful artifact may be a task contract or spec folder: objective, product behavior, technical strategy, files, tests, screenshot checks, blocked conditions, and what the agent must report before ending.
